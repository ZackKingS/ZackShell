## 1. 技术选型

### 1.1 总体方案
采用 **Electron** 作为桌面外壳,主进程负责 SSH/SFTP/监控等后台逻辑,渲染进程负责 UI。该方案开发效率最高,终端生态最成熟(VS Code、Tabby 同款 xterm.js)。

### 1.2 技术栈清单

| 层 | 选型 | 说明 |
|---|---|---|
| 桌面外壳 | Electron | 双进程模型天然契合「UI ↔ 后台」分离 |
| 语言 | TypeScript | 全栈类型安全 |
| UI 框架 | React 18 | 组件化,生态丰富 |
| UI 组件库 | Ant Design | 表格、树、表单、布局开箱即用 |
| 样式 | Tailwind CSS | 快速布局 |
| 终端渲染 | xterm.js + addon-fit / addon-search | VT100 兼容 |
| SSH / SFTP | ssh2(node) | Shell / SFTP / Exec / 端口转发一站式 |
| 实时图表 | uPlot | 轻量高性能,适合流量曲线 |
| 状态管理 | Zustand | 轻量,够用 |
| 凭据存储 | keytar(系统凭据库)+ AES 兜底 | 不明文落盘 |
| 配置存储 | electron-store(JSON) | 会话、设置持久化 |
| 进程通信 | Electron IPC(contextBridge) | 安全暴露 API |
| 构建 | Vite + electron-builder | 开发热更 + 打包 NSIS 安装包 |
| 测试 | Vitest + Playwright(E2E) | 单测 + 端到端 |

### 1.3 备选方案(备查)
- **Tauri + Rust(russh)**:体积约 1/10、内存更低,但 SSH 库资料少、IPC 开发成本高。建议 v1 用 Electron,若后期对体积有强诉求再评估迁移。
- **C# / Avalonia + SSH.NET**:Windows 原生但终端控件生态弱。
- 不选 Java/JavaFX(FinalShell 原方案)与 C++/Qt(WindTerm 方案),个人开发效率低。

## 2. 系统架构

### 2.1 进程划分

```
┌─────────────────── Renderer 进程 (UI / 沙箱) ─────────────────┐
│  React + Ant Design + Zustand                                │
│  ├─ SessionSidebar    会话管理侧栏                            │
│  ├─ TerminalView      终端 (xterm.js)                        │
│  ├─ FileManager       SFTP 文件管理器                        │
│  ├─ MonitorPanel      监控面板 (uPlot 图表 + 表格)           │
│  └─ Settings          设置                                   │
│                  ▲  仅通过 preload 暴露的 window.api 通信     │
└──────────────────┼───────────────────────────────────────────┘
                   │ contextBridge (类型安全 IPC)
┌──────────────────┼───────────── Main 进程 (Node / 后台) ──────┐
│  IpcRouter       接收渲染层请求、转发事件                     │
│  ├─ SessionManager      连接生命周期管理(多连接)            │
│  │    └─ SshConnection  封装单个 ssh2 Client                 │
│  │         ├─ ShellChannel   PTY ↔ 终端字节流                │
│  │         ├─ SftpChannel     文件操作                       │
│  │         └─ ExecChannel     执行监控采集命令               │
│  ├─ MonitorService      定时采集 + 解析 + 推送               │
│  ├─ CredentialStore     keytar / AES 加解密                  │
│  ├─ ConfigStore         electron-store 持久化                │
│  └─ TransferManager     上传下载队列与进度                   │
└───────────────────────────────────────────────────────────────┘
```

### 2.2 关键原则
1. **渲染进程零信任**:`nodeIntegration=false`、`contextIsolation=true`,UI 不直接碰网络 / 文件系统,只通过 preload 暴露的白名单 API。
2. **后台单点管理连接**:所有 SSH 连接由主进程 `SessionManager` 持有,UI 持有 `sessionId` 句柄。
3. **三类数据流分通道**,避免互相阻塞:
   - 终端字节流(高频):`terminal:data`
   - 监控 JSON(低频):`monitor:update`
   - 文件传输进度(事件流):`transfer:progress`

## 3. 模块设计

### 3.1 SessionManager / SshConnection

```ts
interface SessionConfig {
  id: string;
  name: string;
  group?: string;
  host: string;
  port: number;          // 默认 22
  username: string;
  authType: 'password' | 'privateKey';
  // 密码 / 私钥不在此结构中,运行时从 CredentialStore 取
  proxyJump?: string;    // 跳板机会话 id
  keepaliveInterval?: number;
}

class SshConnection {
  connect(cfg: SessionConfig, secret: Secret): Promise<void>;
  openShell(cols, rows): ShellChannel;      // 终端
  openSftp(): Promise<SftpChannel>;         // 文件
  exec(cmd: string): Promise<string>;       // 监控采集
  resize(cols, rows): void;
  dispose(): void;
  on('close' | 'error', handler): void;
}
```

- 基于 `ssh2` 的 `Client`。`openShell` 请求 PTY(`pty: true`),把 `stream` 的 data 通过 IPC 推到渲染层 xterm.js;反向把用户输入写回 stream。
- `keepalive` 通过 ssh2 的 `keepaliveInterval` 配置,断线触发 `close` 事件,通知 UI。
- 跳板机:先连 proxyJump 会话,用其 `forwardOut` 建立到目标的隧道,再在隧道 socket 上建立目标连接。

### 3.2 MonitorService(核心)

**采集策略**:目标机零 agent,通过一条合并命令读取 `/proc`,减少 SSH 往返。

```bash
# 一次性采集脚本(用分隔符拼接,主进程解析)
echo '##CPU##';   cat /proc/stat | grep '^cpu '
echo '##MEM##';   cat /proc/meminfo
echo '##NET##';   cat /proc/net/dev
echo '##DISK##';  df -P -B1
echo '##LOAD##';  cat /proc/loadavg
echo '##UP##';    cat /proc/uptime
echo '##PROC##';  ps -eo pid,comm,%cpu,%mem --sort=-%mem | head -n 15
```

**计算要点(都是差值采样)**:
- **CPU 使用率**:取两次 `/proc/stat` 的 `cpu` 行,计算
  `usage = 1 - Δidle / Δtotal`(total = user+nice+system+idle+iowait+irq+softirq+steal)。
- **网络速率**:两次 `/proc/net/dev` 的某网卡 rx/tx 字节差 ÷ 时间差 = B/s。
- **内存**:`MemTotal - MemAvailable` 为已用;Swap 同理。
- **磁盘**:解析 `df -P -B1`,逐挂载点 used/total。

```ts
class MonitorService {
  start(sessionId: string, intervalMs = 2000): void;  // 启动定时采集
  stop(sessionId: string): void;
  // 内部:每个 tick exec 采集脚本 → 解析 → 与上次采样做差值 → emit('monitor:update', metrics)
}

interface Metrics {
  cpu: { usage: number; cores: number };
  mem: { used: number; total: number };
  swap: { used: number; total: number };
  disks: Array<{ mount: string; used: number; total: number }>;
  net: Array<{ iface: string; rxBps: number; txBps: number }>;
  load: [number, number, number];
  uptimeSec: number;
  procs: Array<{ pid: number; name: string; cpu: number; mem: number }>;
}
```

- **发行版差异兜底**:优先 `/proc`(几乎所有 Linux 通用);`ps`/`df` 字段在 BusyBox 上可能不同,做格式探测与降级。
- 监控用**独立 exec 通道**,不占用交互终端;采集失败不影响终端会话。

### 3.3 SFTP / TransferManager

- 基于 ssh2 SFTP 子系统:`readdir`、`stat`、`fastGet`/`fastPut`(带 `step` 回调出进度)、`mkdir`、`unlink`、`rename`、`chmod`。
- **传输队列**:任务进入队列,串行 / 有限并发执行,`transfer:progress` 事件携带 `{ taskId, transferred, total, speed }`。
- **在线编辑**:下载到本地临时文件 → 用内置编辑器(Monaco)打开 → 保存时回传 → 清理临时文件。
- 大文件用流式,避免一次性读入内存。

### 3.4 CredentialStore(安全)

- 首选 `keytar`,把密码 / 私钥 passphrase 写入 **Windows 凭据管理器**,以 `zackshell:<sessionId>` 为键。
- 兜底:用户设主密码 → PBKDF2 派生密钥 → AES-256-GCM 加密后存 electron-store。
- **主机密钥校验**:维护 `known_hosts`,首次连接提示指纹,变更时告警(防中间人)。
- 配置文件中**绝不**出现明文密码 / 私钥。

### 3.5 ConfigStore

- `electron-store` 存:会话列表(不含密钥)、分组、外观设置、采集频率等。
- 路径:`%APPDATA%/ZackShell/config.json`。
- 提供导入 / 导出(脱敏,不含凭据)。

### 3.6 IPC 协议(渲染 ↔ 主)

通过 preload `contextBridge` 暴露类型化 API,渲染层调用 `window.api.xxx`:

| 通道 | 方向 | 用途 |
|---|---|---|
| `session:connect` | R→M(invoke) | 建立连接,返回 sessionId |
| `session:disconnect` | R→M | 断开 |
| `terminal:input` | R→M | 用户键入 |
| `terminal:data` | M→R(event) | 终端输出 |
| `terminal:resize` | R→M | 尺寸变化 |
| `sftp:list` / `sftp:upload` / `sftp:download` / ... | R→M | 文件操作 |
| `transfer:progress` | M→R(event) | 传输进度 |
| `monitor:start` / `monitor:stop` | R→M | 控制采集 |
| `monitor:update` | M→R(event) | 监控数据推送 |
| `session:closed` | M→R(event) | 连接断开通知 |

## 5. 数据流示例

### 5.1 终端输入输出
```
用户键入 → xterm.onData → window.api.terminalInput(sessionId, data)
        → IPC terminal:input → ShellChannel.stream.write(data)
远端输出 → stream.on('data') → IPC terminal:data → xterm.write(data)
```

### 5.2 监控刷新
```
MonitorService tick(2s) → ExecChannel.exec(采集脚本)
   → 解析文本 → 与上次采样差值计算 → Metrics
   → IPC monitor:update → MonitorPanel 更新 uPlot 曲线 / 表格
```

## 6. 关键技术难点与对策

| 难点 | 对策 |
|---|---|
| 终端尺寸与 TUI 程序兼容 | 用 addon-fit 计算 cols/rows,resize 时同步 `stream.setWindow`;请求 PTY 时指定 `term: 'xterm-256color'` |
| 中文 / 宽字符 | xterm.js 启用 Unicode11 addon;PTY 用 UTF-8 |
| CPU/网络瞬时值 | 必须两次采样做差值,首帧无数据只缓存不展示 |
| 发行版命令差异 | 优先 /proc;对 df/ps 做字段容错与降级解析 |
| 大文件传输不卡 UI | 主进程流式传输 + 节流的进度事件(如 200ms 一次) |
| 安全(渲染逃逸) | contextIsolation + 无 nodeIntegration + 最小白名单 API |
| 凭据泄露 | 系统凭据库 / 主密码加密;配置脱敏 |
